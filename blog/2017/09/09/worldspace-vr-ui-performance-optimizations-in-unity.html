One of the largest challenges so far while developing Skytropolis has been solving the best ways to keep performance smooth. We need to hit and keep the game at over 90FPS for the Vive & Rift, while maintaining a visually interesting aesthetic.

One of the things we noticed was that performance was dropping after we rolled out a new update to this card UI. Each time a card category with more than 10+ cards was chosen, the frame rate would drop to around 50 - 60 FPS.

Digging in to Unityâ€™s profiler showed that re-draws of the UI each time the player moved their hand was the issue.

Each element in the tree above is a GameObject. And in typical Unity fashion, each GameObject has a Transform, which controls itâ€™s position, rotation and scale in the world. When the card hand is open, and the player moves their hand the VRCardUISystem loosely follows the hand around so the UI is always available and ready to use.

This turned out to be the biggest issue with the cards. As the VRCardUISystem Transform follows the playerâ€™s hand, and their hand moves every single frame (even miniscule amounts are picked up by the VR sensors) under the hood Unity has to tell every child Transform what itâ€™s new position is, move it there, then send that to the graphics pipeline for rendering to the playerâ€™s headset screens.

After a few various attempts at improving performance via other means, such as changing the VRCardUISystemâ€™s update rate to not happen every frame, and using object pooling instead of Instantiate() and Destroy() we found the performance issue still existed.

The solution pt 1: a much simpler hierarchy.

The solution was to simplify. I had read somewhere that Unity prefers very simple hierarchy, and instead of grouping objects together logically (which is easier for the developers and level editors to consume), it ideally wants every GameObject (and Transform) to be at the root level. This is a mess for us developers to read, but helps itâ€™s internal systems.

The solution pt 2: One background image instead of multiple Image components.

On top of removing the container groups, we decided about half of the Unity Image components could be baked into one PNG background sprite in Photoshop, as could the Demands Images and Services Arrows (showing the player how this building will affect their cityâ€™s demands, surplus and service import / export costs). This resulted in each card only having 20 child GameObjects (and Transforms) instead of 59. And when you have 20 cards in your hand, that means only 400 GameObjects (and Transforms) instead of 1,180!
